# SuperHermes — 基于 RAG 的企业级知识库问答系统

> 全栈独立开发 | Python + FastAPI + LangChain + Milvus + Vue 3

---

## 项目简介

开发了一个完整的 RAG（检索增强生成）知识库问答系统，支持 PDF/Word/Excel 文档上传、混合向量检索、流式 AI 对话，以及实时思考过程可视化。系统采用 LangChain Agent + LangGraph 架构，实现从文档解析到答案生成的全链路自动化。

---

## 技术栈

- **后端**：FastAPI、LangChain、LangGraph、SQLAlchemy、Pydantic
- **向量检索**：Milvus（HNSW + 稀疏倒排索引）、手写 BM25、RRF 融合排序、Jina Rerank
- **Embedding**：本地 `BAAI/bge-m3`（HuggingFace），1024 维稠密向量
- **数据库/缓存**：PostgreSQL 15、Redis 7
- **前端**：Vue 3（CDN SPA）、SSE + ReadableStream 流式渲染
- **部署**：Docker Compose（PostgreSQL + Redis + Milvus + MinIO + etcd）

## 核心工作

### 1. 混合检索 RAG Pipeline

- 设计并实现 **Dense + Sparse 混合检索**，稠密向量使用 `bge-m3`，稀疏向量手写 BM25 算法（中英混合分词），通过 Milvus `AnnSearchRequest` + `RRFRanker(k=60)` 融合两路召回，免调参
- 接入 **Jina Rerank API** 实现精排，支持双向降级（Hybrid 失败 → 纯 Dense → RRF 兜底），保证检索稳定性
- 基于 **LangGraph** 构建四节点状态图：`retrieve_initial → grade_documents → rewrite_question → retrieve_expanded`，通过 LLM 结构化输出进行相关性打分门控
- 实现 **查询重写路由**，由 LLM 自主选择 Step-Back / HyDE / Complex 策略，解决初次检索不匹配的问题

### 2. 三级分块与 Auto-merging 检索

- 设计 **L1/L2/L3 三级滑动窗口分块**策略（≈1200/600/300 token），每层携带 `chunk_id`、`parent_chunk_id`、`root_chunk_id` 层级元数据
- 采用 **Leaf-only 存储**：仅 L3 叶子块写入 Milvus，L1/L2 父块存入 PostgreSQL DocStore，减少向量冗余
- 实现 **两段 Auto-merging**（L3→L2→L1）：同一父块下 ≥ 2 个子块命中时自动替换为父块，兼顾检索精准度与上下文完整性

### 3. 实时思考链路可视化

- 解决 **同步工具阻塞异步事件循环** 的核心难题：通过 `call_soon_threadsafe` 将 LangChain 线程池中的 RAG 步骤安全调度到主 asyncio Loop，实现检索/评分/重写各阶段实时推送到前端
- 前端使用 **SSE + ReadableStream** 手动解析事件流，维护微型状态机（Idle → Thinking → Active RAG → Streaming），思考气泡与回答文本在同一气泡内无缝切换，无 UI 抖动
- 完整记录 `rag_trace`（评分结果、路由决策、重写策略、检索分数、Rerank 分数），前端可展开查看 RAG 全过程

### 4. 流式输出与确定性中断

- 基于 `agent.astream(stream_mode="messages")` 实现逐 token 推送，通过统一 `asyncio.Queue` 汇聚内容 chunk 与 RAG 步骤事件
- 实现 **确定性中断止损**：前端 `AbortController.abort()` → 后端显式捕获 `GeneratorExit` 并 `task.cancel()`，注入 `CancelledError` 关闭 TCP 连接，服务端立即停止推理，节省 Token

### 5. BM25 统计持久化与一致性

- 实现 BM25 词表 `vocab`、文档频次 `doc_freq`、总文档数 `total_docs` 持久化至 JSON，采用 **原子写盘**（tmp → replace）防止中断损坏
- 文档入库时 `increment_add`、删除/覆盖上传前先从 Milvus 拉取全文再 `increment_remove`，保证 BM25 统计与向量库 **严格一致**
- 词表下标不回收，避免与历史稀疏向量维度冲突

### 6. 认证体系与性能优化

- 实现 JWT Bearer Token 认证 + RBAC 权限控制（admin 管理文档，user 仅聊天），PBKDF2-SHA256 密码存储，兼容历史 bcrypt 平滑迁移
- 引入 **Redis 缓存**会话列表、消息列表与父文档，写入/删除后主动失效
- 聊天历史超过 50 条触发 **LLM 摘要压缩**，保留关键上下文并控制 Token

## 项目亮点（面试可展开）

1. **混合检索工程化落地**：不是简单调用 API，而是手动实现 BM25 稀疏向量生成、状态持久化、增量更新、删除同步，与 Milvus Hybrid Search 深度集成
2. **跨线程事件调度**：解决了 Python 异步 Web 服务中"同步工具阻塞异步循环"的经典难题，让 RAG 检索过程在模型思考阶段实时可见
3. **确定性资源回收**：不依赖框架自动取消，通过显式捕获 `GeneratorExit` + `task.cancel()` 确保客户端断开时立刻止损
4. **完整的 RAG 可观测性**：每个阶段（检索、评分、重写、合并）都发射结构化事件，前端可视化，后端记录 `rag_trace`，便于调试和优化
5. **全栈独立开发**：从后端 API 设计、向量检索策略、异步流式架构，到前端 SSE 解析、状态机 UI、CSS 样式，以及 Docker 基础设施编排，全部独立完成

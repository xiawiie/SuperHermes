## 五、核心链路

### 5.1 文档入库链路

```
上传 PDF/Word/Excel
  → 检测同名文件：分页拉取旧 chunk 全文 → BM25 increment_remove → 删除旧向量
  → 三级滑动窗口分块（L1≈1200 → L2≈600 → L3≈300 token）
  → L1/L2 父块 → PostgreSQL DocStore（parent_chunk_store）
  → L3 叶子块 → BM25 increment_add → 生成 Dense + Sparse 向量 → 写入 Milvus
```

**关键设计**：

- **覆盖上传先删后写**，保证 BM25 统计与向量库一致
- **`query_all` 分页拉取**（单次 limit 受 16384 约束），避免 RPC 报错
- **词表下标不回收**，防止与历史稀疏向量维度冲突
- **原子写盘**（tmp → replace），避免中断导致状态文件损坏

### 5.2 RAG 检索链路（LangGraph 状态图）

```
retrieve_initial
  →  Milvus Hybrid Search（Dense + Sparse，RRF k=60）
  →  Jina Rerank 精排（可选）
  →  Auto-merging（L3→L2→L1，阈值≥2 个子块触发）
  │
grade_documents（LLM 结构化输出 yes/no）
  ├─ yes → 直接进入生成
  └─ no → rewrite_question
        → LLM 路由选择策略：
          · step_back：抽象为退步问题 + 回答，用于细节类查询
          · hyde：生成假设性文档，用于概念类查询
          · complex：同时执行 step_back + hyde，用于复杂查询
        → retrieve_expanded（二次召回 + 去重）
        → 生成
```

### 5.3 流式输出 + 实时思考可视化

```
chat_with_agent_stream()
  ├── 创建统一 output_queue (asyncio.Queue)
  ├── 设置 _RagStepProxy → emit_rag_step 直接入队
  ├── _agent_worker 后台任务：
  │     agent.astream(stream_mode="messages")
  │       ├── AIMessageChunk → {"type": "content"} 入队
  │       └── tool_call_chunks → 跳过
  │
  └── 主循环：await output_queue.get() → yield SSE
        ▲
        │ (并发) RAG 工具在线程池中执行
        │ emit_rag_step() → call_soon_threadsafe → 入队
        │ {"type": "rag_step"} 实时推送，不等 agent 产出
```

**核心突破**：`call_soon_threadsafe` 解决同步工具阻塞异步事件循环的问题，实现检索步骤在模型"思考中"阶段实时推送到前端。

### 5.4 会话记忆链路

```
每轮问答 → PostgreSQL（按用户 + session 隔离）
  → 消息 > 50 条 → LLM 摘要压缩，保留关键上下文
  → Redis 缓存会话列表与消息列表，加速历史回放
  → 写入/删除后主动失效缓存，保证一致性
```

## 六、工程亮点

### 6.1 混合检索 + 双向降级

| 路径 | 说明 |
|------|------|
| **主路径** | Hybrid（Dense + Sparse + RRF）→ Rerank → Auto-merge |
| **降级 1** | Hybrid 失败 → 纯 Dense 检索 |
| **降级 2** | Rerank 不可用/报错 → RRF 结果直接返回 |
| **降级 3** | 全部失败 → 返回空，Agent 如实告知 |

MilvusManager 采用**每次操作新建客户端 + 自动重试**模式，避免跨线程/跨请求的客户端失效。

### 6.2 BM25 状态持久化

- 词表 `vocab` + 文档频次 `doc_freq` + 总文档数 `total_docs` + 总 token 长度 `sum_token_len`
- 写入时 `increment_add`，删除时 `increment_remove`，原子写盘（tmp → replace）
- 检索与写入共享同一 `embedding_service` 单例，避免状态分裂

### 6.3 确定性中断止损

- 前端 `AbortController.abort()` → FastAPI 抛出 `GeneratorExit`
- 显式捕获并 `agent_task.cancel()`，注入 `CancelledError`
- 不依赖框架级联取消，确保真正关闭 TCP 连接，节省服务端 Token

### 6.4 安全与权限

- JWT Bearer Token 认证，PBKDF2-SHA256 密码存储，兼容历史 bcrypt
- RBAC：admin 管理文档，user 仅聊天
- 会话按用户隔离，Redis 缓存写入/删除后主动失效

### 6.5 三级分块 + Auto-merging

- **L1（≈1200 token）**：根级分块，保留最大上下文
- **L2（≈600 token）**：中间层，承上启下
- **L3（≈300 token）**：叶子层，仅写入 Milvus，检索精准度高
- **Auto-merging**：L3 召回后，同一父块下 ≥ 2 个子块命中时，自动替换为 L2 或 L1 父块，两段合并（L3→L2→L1）

### 7.3 Token 消耗

| 指标 | 数值 | 占比 |
|------|------|------|
| 平均总 Token / 样本 | 4,778.60 | 100% |
| 平均输入 Token / 样本 | 3,545.33 | 74.19% |

### 7.4 评估解读

**质量**：15/15 满分，双评估器完全一致。在当前测试集上回答准确性无可挑剔。

**时延**：平均 27s 偏长，波动范围 18s ~ 48s（2.65 倍差距）。差距来源：
- 触发重写检索（二次召回 + 额外 LLM 调用）
- Rerank API 外部调用增加网络延迟
- Step-Back/HyDE 策略需要额外 LLM 推理（退步问题 + 回答 / 假设性文档）

**Token**：输入占 74%，是 RAG 场景典型结构。优化方向包括检索结果去重精简、减少不必要的相关性打分 LLM 调用。

## 八、综合 SWOT 分析

### 优势 (Strengths)

- RAG 架构完整：检索 → 评分 → 重写 → 二次检索，全链路可观测
- 混合检索 + RRF 免调参融合，兼顾语义与词匹配
- 实时思考链路可视化，用户体验优秀
- BM25 状态持久化，重启不丢失 IDF 统计
- 确定性中断止损，客户端断开即节省服务端 Token
- Milvus 客户端自动重连，鲁棒性强
- 前端零构建工具，部署极简

### 劣势 (Weaknesses)

- 时延偏高，最大 48s 影响用户体验
- 输入 Token 占 74%，LLM 调用成本可优化
- 纯 CDN 前端难以应对大规模 UI 扩展
- 样本量仅 15 条，评估统计效力有限
- 缺少单元测试/集成测试覆盖
- 前端无类型检查，维护成本随规模增长

### 机会 (Opportunities)

- 多 Agent 拆分工具，提升工具选择准确性
- 搭建 RAG 评估体系，量化优化效果
- 多模态 Embedding 支持图片/表格检索
- SQL Assistant 等新 Skill 扩展场景
- 死循环检测与自动恢复机制

### 威胁 (Threats)

- LLM 上下文窗口持续扩大可能削弱分块策略价值
- BM25 手写分词对复杂中文支持有限
- 过度依赖外部 LLM API，断网即停摆
- Milvus 运维成本较高，小规模场景可选更轻量方案
- 无 CI/CD 流水线，手动部署易出错

## 九、优化建议

| 优先级 | 方向 | 具体措施 |
|---|---|---|
| **P0** | 评估覆盖 | 样本量从 15 扩大到 50-100 条，覆盖边界场景（无匹配文档、多文档冲突、复杂多步问题） |
| **P0** | 时延优化 | 对简单问题跳过 RAG pipeline 直接回答；缓存高频问答对；Rerank 与 grading 并行化 |
| **P1** | Token 成本 | 检索结果去精后再送入 LLM；减少 grading 调用（仅在候选数 > 阈值时触发） |
| **P1** | 测试覆盖 | 为核心链路（分块→入库→检索→合并）添加单元测试 |
| **P2** | BM25 分词 | 中文引入 Jieba/HanLP 替代单字切分，提升稀疏检索精度 |
| **P2** | 前端工程化 | 迁移到 Vite + TypeScript，支持组件化和路由 |
| **P3** | CI/CD | 建立 GitHub Actions 流水线：lint → test → build → deploy |
